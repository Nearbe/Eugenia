## llama.cpp — что такое "архитектуры"

"63+ архитектур" — это **типы нейросетей**, которые поддерживает llama.cpp. Каждая архитектура = другой формат модели:

### Основные архитектуры:
- **LLaMA/LLaMA 2/LLaMA 3** — Meta
- **Qwen** — Alibaba (твоя модель!)
- **Mistral/Mixtral** — Mistral AI
- **Gemma** — Google
- **Deepseek** — DeepSeek
- **Phi** — Microsoft
- **Falcon** — TII
- **Bloom**, **GPT-2**, **StableLM**, **Yi**, **OLMo**, **Granite** и т.д.

**Суть:** llama.cpp — это универсальный inference engine. Он умеет загружать и выполнять модели любого из этих типов. Но для этого модели должны быть в формате **GGUF**.

## llama-server — ключевые фичи

### Что делает llama-server:
HTTP-сервер, который принимает запросы по OpenAI-compatible API и возвращает генерацию от модели.

### Ключевые фичи сервера:

#### 1. API Endpoints
| Endpoint | Описание |
|----------|----------|
| `POST /v1/chat/completions` | OpenAI Chat Completions API |
| `POST /v1/completions` | OpenAI Completions API |
| `POST /v1/responses` | OpenAI Responses API |
| `POST /v1/embeddings` | OpenAI Embeddings API |
| `POST /v1/messages` | Anthropic Messages API |
| `POST /v1/messages/count_tokens` | Подсчёт токенов |
| `GET /v1/models` | Информация о модели |
| `GET /slots` | Состояние слотов (контекстов) |
| `GET /metrics` | Prometheus метрики |
| `GET /props` | Глобальные свойства сервера |

#### 2. Streaming
- Все endpoints поддерживают streaming (SSE — Server-Sent Events)
- Данные текут потоком, не накапливаются
- Идеально для твоего сценария: "через stream натурально будут течь данные"

#### 3. Continuous Batching
- Модель обрабатывает несколько запросов одновременно
- Каждый запрос в своём "слоте"
- Слоты можно сохранять/восстанавливать/стирать

#### 4. Speculative Decoding
- Ускорение генерации через speculative decoding
- Маленькая модель помогает большой модели генерировать быстрее

#### 5. Function Calling / Tool Use
- OpenAI-style function calling
- Anthropic-style tool use
- Требуется флаг `--jinja`

#### 6. Schema-constrained JSON
- Генерация валидного JSON по схеме
- `response_format: {type: "json_object", schema: {...}}`

#### 7. Grammar-constrained Output
- GBNF грамматики для строгого формата вывода
- JSON grammar, regex grammar, custom grammar

#### 8. Multimodal Support
- Загрузка изображений через `image_url`
- Base64 или remote URL
- Для Qwen3.6-35B-A3B: мульти模дальная версия (есть mmproj файл)

#### 9. Multi-Model Router
- Один сервер может управлять несколькими моделями
- Динамическая загрузка/выгрузка моделей
- `POST /models/load`, `POST /models/unload`
- Автоматическая загрузка при первом запросе

#### 10. Sleeping on Idle
- Автоматический sleep при бездействии
- Модель выгружается из RAM
- Новый запрос автоматически загружает модель

#### 11. KV Cache Management
- Слоты: `POST /slots/{id}?action=save|restore|erase`
- Можно сохранить контекст слота в файл
- Восстановить из файла
- Стереть

#### 12. LoRA Adapters
- Динамическая загрузка LoRA адаптеров
- `GET /lora-adapters`, `POST /lora-adapters`
- Масштабирование каждого адаптера

#### 13. Monitoring
- `/metrics` — Prometheus метрики
- `/slots` — состояние каждого слота
- Промпт-токены, предсказанные токены, скорость, timing

#### 14. Web UI
- Встроенный веб-интерфейс
- Настраиваемые предпочтения через `--webui-config`

## Твоя модель: Qwen3.6-35B-A3B

### Архитектура:
- **MoE (Mixture of Experts)**: 35B параметров всего, но только 3B активны на каждый токен
- Это значит: быстрая генерация, потому что активна только малая часть модели
- 4 shard-а (модель разделена на 4 части)

### Какая версия для llama-server?
- **GGUF** — да, llama-server работает с GGUF напрямую
- **MXFP4 (safetensors)** — нет, llama-server не работает с safetensors. Нужна конвертация в GGUF

## Сетевой сетап — ответ на вопросы

### 1. Где будет работать модель?
✅ **Локально** на MacBook Pro M4 Max, 128GB RAM

### 2. Какая модель?
✅ **Qwen3.6-35B-A3B** (MoE, 3B active per token)
- GGUF версия: `Qwen3.6-35B-A3B-Q4_K_M.gguf` (21.2 ГБ)
- Мультимодальная: есть mmproj файл

### 3. Какой API-формат?
✅ **OpenAI-compatible** (базовый), но **потребует модификации**
- Базовый `/v1/chat/completions` — работает из коробки
- Модификации — пока не ясно какие, нужно обсудить

### 4. Как модель будет запущена?
✅ **Через llama-server** (из llama.cpp)
- Вопрос "почему возник" — потому что ты сам сказал что llama-server обсуждаем, но я задал вопрос как формальность. Ты прав, этот вопрос должен был отпасть сам.

### 5. Проблемы с конфигурацией?
> "Модель может генерировать очень быстро, но есть узкие места с выполнением, что приводят к осложнениям при работе с большими файлами или при попытке генерировать большой документ. Это всё в первую очередь должно работать на максимальной скорости за счёт связки всего этого, где через stream натурально будут течь данные, но не накапливаться бесконечно, а уничтожаться за ненадобностью. И только 5% от полезной информации должно оседать."

**Ключевые проблемы:**
- Большие файлы → узкие места
- Большой документ → узкие места
- Данные должны течь через stream, не накапливаться
- Только 5% информации должно оседать (остальное уничтожаться)

## Главный процесс работы — что должно работать

На основе твоих слов:

### 1. Модель (llama-server) → генерирует токены
- GGUF модель загружена в RAM
- Metal framework для GPU acceleration
- Streaming через SSE

### 2. Данные текут streamом
- Чанки генерации приходят по одному
- Не накапливаются — обрабатываются и уничтожаются
- Только 5% полезной информации сохраняется

### 3. Rolling window контекста
- Я вижу одно — ты видишь то же самое
- Контекст открыт и прям
- Управление контекстом строго через UI

### 4. UI — критичная зона ответственности
- Прямое управление контекстом
- Изменение контекста во время исполнения
- Область максимальной зоны

### 5. MCP JetBrains тулзы
- Вместо встроенных тулзов qwen-code
- "Я тебе говорю используй, а ты раз используешь и всё, больше не используешь"
- Нужно чтобы тулзы работали непрерывно

### 6. LSP-интеграция
- Возможно вместо отдельного telemetry
- Мониторинг работы через LSP
- Данные о том как ТЫ работаешь, чтобы ТЫ мог видеть проблемы
