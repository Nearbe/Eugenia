# Эксперименты: виртуальный контейнер и подача данных

## Контекст
- Eugenia.py = виртуальный контейнер (не Docker, не процесс, а само исполнение)
- 421GB виртуальной памяти = устойчивая платформа вычислений
- Привязка к полу-тактам = 100% предсказуемость
- GIL Python = детерминированный поток
- IEEE 754 ограничения = преимущество при грамотном подходе
- М4 Max: 128GB unified memory, 40-core GPU, Neural Engine

## Что известно
1. Рекурсивный цикл работает и вычисляется с бешеной скоростью
2. Виртуальная память = платформа, не просто резерв
3. Физически аллоцируется всегда (421GB виртуальной = устойчивая платформа)
4. Python идеален для хитростей из-за GIL и интерпретатора
5. IEEE 754 ограничения можно использовать на пользу

## Что НЕ известно (нужны эксперименты)
1. Какой принцип засовывания данных? (помаленьку / блоками)
2. Куда именно засовывать? (mmap? shared memory? что?)
3. Как измерять скорость правильно?
4. Как связать веса модели с этой платформой?
5. Что даёт GPU в этом контексте?

## Эксперимент 1: Принцип засовывания данных

### Цель
Понять: помаленьку (stream) или блоками (batch) работает лучше.

### Подход
1. Создать тестовый массив данных (1GB, 10GB, 50GB)
2. Загрузить помаленьку (chunk by chunk)
3. Загрузить блоком (all at once)
4. Измерить время доступа к данным
5. Измерить throughput

### Метрики
- Время аллокации
- Время первого доступа
- Время последовательного чтения
- Время случайного чтения
- Throughput (GB/s)

### Инструменты
- `mmap` — memory-mapped files
- `shared_memory` — shared memory
- `numpy` — массивы
- `perf_counter` — тайминг

### Ожидаемый результат
- Понимание какого подхода держать
- Конкретные цифры throughput

## Эксперимент 2: Куда засовывать данные

### Цель
Понять какой механизм аллокации работает лучше для виртуального контейнера.

### Варианты
1. **mmap файл** — файл в filesystem, mapped в memory
2. **shared_memory** — разделяемая память между процессами
3. **numpy arrays** — стандартные массивы
4. **ctypes** — прямая работа с памятью
5. **memoryview** — view на буфер без копирования

### Подход
Для каждого варианта:
1. Аллоцировать 1GB
2. Записать данные
3. Прочитать данные
4. Измерить время
5. Измерить потребление

### Ожидаемый результат
- Лучший механизм для виртуального контейнера
- Понимание trade-offs каждого подхода

## Эксперимент 3: Связь с весами модели

### Цель
Понять как засунуть веса модели в виртуальный контейнер.

### Варианты
1. **GGUF файл → mmap** — загрузить GGUF через mmap
2. **safetensors → mmap** — загрузить safetensors через mmap
3. **Прямое чтение из GGUF** — через llama.cpp C API
4. **Python-парсинг GGUF** — прочитать и аллоцировать в numpy

### Подход
1. Взять Qwen3.6-35B-A3B-Q4_K_M.gguf (21.2GB)
2. Загрузить каждым из 4 способов
3. Измерить время загрузки
4. Измерить время доступа к весам
5. Сравнить

### Ожидаемый результат
- Понимание как загружать веса в Eugenia.py
- Лучший способ для этой архитектуры

## Эксперимент 5: IEEE 754 как преимущество

### Цель
Понять как использовать IEEE 754 ограничения на пользу.

### Варианты
1. **Inf/NaN как маркеры** — использовать для управления потоком
2. **Переполнение как триггер** — overflow = событие
3. **Denormal numbers** — поднормальные числа для тонкой настройки
4. **Rounding modes** — разные режимы округления

### Подход
1. Создать данные которые вызывают Inf/NaN
2. Поймать и использовать эти события
3. Измерить overhead обработки
4. Определить практическое применение

### Ожидаемый результат
- Понимание как IEEE 754 работает на нас
- Конкретные паттерны использования
