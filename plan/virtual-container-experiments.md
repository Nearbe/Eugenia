# Эксперименты: виртуальный контейнер для llama-server

## Контекст
Пользователь хочет свой минимальный аналог Docker — виртуальное окружение для запуска llama-server.
Это окружение даёт Eugenia.py.

## Цели экспериментов
1. Понять какие возможности нужны от контейнера
2. Понять как Eugenia.py управляет этим окружением
3. Определить минимальный набор изоляции
4. Найти подход, который работает на macOS (M4 Max, 128GB RAM)

## Ключевые вопросы для экспериментов

### Эксперимент 1: Что значит "минимальные максималки"?
**Цель:** Определить баланс между минимальным размером и максимальной функциональностью.

**Вопросы:**
- Контейнер должен быть самодостаточным (всё внутри) или подключаться к внешнему?
- Нужна ли файловая изоляция?
- Нужна ли сетевая изоляция?
- Нужна ли процессная изоляция?
- Размер контейнера важен? (сейчас llama.cpp ~487МБ без моделей)

**Подход:**
- Протестировать запуск llama-server без контейнера (просто процесс)
- Протестировать запуск в Docker (если работает на macOS)
- Сравнить с идеей "своего контейнера"

### Эксперимент 2: Почему не Docker?
**Цель:** Понять причины отказа от Docker.

**Вопросы:**
- Docker не работает на macOS? (работает, но через VM)
- Docker слишком тяжёлый?
- Docker конфликтует с чем-то?
- Или идея "своего" контейнера — философская, а не техническая?

**Подход:**
- Проверить работает ли Docker Desktop на macOS
- Проверить производительность Docker на macOS
- Сравнить с нативным запуском

### Эксперимент 3: Как Eugenia.py управляет контейнером?
**Цель:** Понять связь между Eugenia.py и окружением.

**Вопросы:**
- Eugenia.py запускает llama-server как subprocess?
- Eugenia.py управляет жизненным циклом (старт/стоп)?
- Eugenia.py читает метрики из контейнера?
- Eugenia.py передаёт данные в контейнер?

**Подход:**
- Изучить Eugenia.py и generate.py
- Понять как сейчас запускается/inferenced
- Определить что нужно добавить

### Эксперимент 4: Принцип работы виртуального контейнера
**Цель:** Определить архитектуру контейнера.

**Варианты:**
1. **Простой процесс** — просто subprocess с ограниченным окружением (env, cwd, limits)
2. **jail/chroot** — файловая изоляция (Unix-механизмы)
3. **cgroups** — ограничение ресурсов (CPU, RAM, I/O)
4. **macOS sandbox** — macOS-специфичная изоляция
5. **Минимальный Docker-like** — свой рантайм на Python

**Вопрос:** Какой подход ближе к "минимальные максималки"?

## Важные замечания
- macOS M4 Max — специфичная платформа
- 128GB RAM — много памяти, но нужно эффективно использовать
- KV cache = бесконечно долго — нужно понять что это значит для контейнера
- Multi-model router — контейнер должен поддерживать несколько моделей
