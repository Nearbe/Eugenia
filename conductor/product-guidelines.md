# Product Guidelines: Eugenia

## 1.0 Writing Principles
**Главный принцип:** Краткость — основа ясности. Чем меньше символов для изложения сути, тем лучше. Избегайте воды, лишних описаний и шаблонных фраз.

## 2.0 Language Policy
- **Сложные концепции** (математика, топология, алгоритмы) — на русском языке
- **Базовые понятия** (API, конфигурация, установка) — на доступном языке, допустим английский
- **Запрещено:** англицизмы, даже устоявшиеся технические термины (например, использовать "сущность" вместо "entity", "контекст" вместо "context" в разговорной речи)
- **Код и комментарии:** допустим английский для универсальности, русский для объяснения сути алгоритмов

## 3.0 Documentation Standards

### 3.1 Comments
- Объяснять **почему** ( rationale ), а не **что** делает код
- Короткие комментарии предпочтительны длинным описаниям
- Русские комментарии допустимы для объяснения алгоритмической логики

### 3.2 Naming Conventions — Process-Driven Decision
**Нейминг — критический процесс принятия решений.** Для каждого нового имени (переменной, функции, класса, модуля):

1. **Анализ контекста:** Определить, что именно реализуется и какова роль сущности
2. **Генерация вариантов:** Предложить 3-4 варианта наименования
3. **Сравнение:** Оценить каждый вариант по критериям:
   - Точность отражения сути
   - Краткость (минимум символов)
   - Отсутствие англицизмов и кальки
   - Соответствие домену проекта
4. **Выбор:** Выбрать лучший вариант из предложенных
5. **Очевидные случаи:** Пропускать процесс только если имя однозначно и не вызывает сомнений. **Но помните:** то, что очевидно для разработчика, может быть неочевидным для модели.

**Пример:**
```
Реализуем: функция, которая преобразует пиксель в дельта-поле

Варианты:
  1. pixel_to_delta() — точный, но длинный
  2. map_pixel() — краткий, но теряется домен
  3. to_delta_field() — явный домен, средний размер
  4. deltaize() — краткий, но англицизм

Выбор: to_delta_field() — баланс точности и ясности
```

### 3.3 Structure
- Использовать иерархические заголовки для навигации
- Списки предпочтительны абзацев
- Таблицы предпочтительны длинных описаний параметров

## 4.0 Code Quality
- **Linting:** ruff (line-length = 100)
- **Typing:** mypy strict mode
- **Форматирование:** собственный `Formatter` (проект в `Formatter/`)
  - Специфика проекта требует кастомного форматера вместо `ruff format`
  - План создания Formatter — отдельный track в инициативе
  - Использовать `cosmic_formatter` как базовую реализацию
  - Конфигурация: `Formatter/pyproject.toml`

## 5.0 Commit Policy
- Фиксировать изменения **после каждой завершённой задачи**
- Сообщение коммита должно отражать суть изменений, а не техническую деталь
