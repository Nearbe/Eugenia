The implementation (`src`) is designed as a "Universal Geometric Knowledge Machine," mapping mathematical theory from the `Universe` documentation into concrete computational modules. The system transforms any input data into geometric invariants that can be stored, queried, and synthesized in a pattern graph.

### Dependency Flow
1.  **Data Ingestion**: Raw data $\rightarrow$ **Universal Normalizer** $\rightarrow$ Normalized Vector $[0, 1]$.
2.  **Geometric Extraction**: Normalized Vector $\rightarrow$ **Binary Sweep** $\rightarrow$ **Jump Detector** $\rightarrow$ **Topology Extractor**.
3.  **Pattern Creation**: Geometric features $\rightarrow$ **Pattern Fusion** $\rightarrow$ $144$-dim `GeometricProfile`.
4.  **Knowledge Synthesis**: `GeometricProfile` $\rightarrow$ **Pattern Graph Engine** (Automatic linking/strengthening) $\rightarrow$ **Runtime Layer** (Querying/Generation).

## 2. Testing Strategies & Project Conventions

### Primary Testing Goals
The testing suite is focused on ensuring that the code implementation adheres strictly to mathematical axioms and "U-system" contracts.

### Key Patterns Observed
1.  **Mathematical Invariant Verification**:
    *   Ensuring inverse operations are consistent (e.g., $H(D(x)) \approx x$).
    *   Verified across multiple types: scalars, lists, and complex vectors.
2.  **Contract Enforcement (Edge Cases)**:
    *   Testing "zero-division contracts" to ensure the system's stability when encountering mathematical singularities (e.g., `safe_divide` handling $x/0$).
    *   Ensuring sentinel values (like `LOG_NEGATIVE_INFINITY`) are handled correctly in logarithmic mappings.
3.  **Precision Management**:
    *   Extensive use of `pytest.approx` to accommodate floating-point precision issues inherent in geometric and topological calculations.
